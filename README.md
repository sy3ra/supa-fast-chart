# Supa-Fast-Chart (고성능 실시간 암호화폐 차트 엔진) 포트폴리오

## 📌 1. 프로젝트 요약 (Project Overview)
- **프로젝트 명:** Supa-Fast-Chart
- **소개:** HTML5 Canvas와 WebAssembly(Rust)를 활용하여 개발한 **초고성능 실시간 가상화폐 차트 웹 애플리케이션**입니다. 브라우저의 DOM 렌더링 및 JS 엔진의 한계를 극복하기 위해 핵심 데이터 연산과 상태 관리를 Rust 엔진으로 위임하여, 거대한 캔들스틱 데이터와 그리기 도구를 프레임당 처리속도 1ms 미만으로 부드럽게 렌더링합니다.
- **주요 스택:**
  - **Frontend:** Vanilla JavaScript, HTML5 Canvas, Vite, CSS
  - **Core Engine (Wasm):** Rust (`wasm-bindgen`, `serde`)
  - **Data Source:** Binance REST API, Binance WebSocket API

## 🛠️ 2. 핵심 기능 및 기술적 도전 (Key Features & Problems Solved)

### 🔥 1) WebAssembly를 통한 렌더링 병목 현상 및 GC 지연 시간 완전 해결
- **문제:** 극초기 구상 단계에서 이 차트 엔진을 **순수 Vanilla JS** 만으로 구현했을 때, 수만 개의 캔들 데이터와 사용자의 커스텀 드로잉(추세선 등) 시각 좌표를 매 프레임 재계산하는 과정에서 치명적인 퍼포먼스 문제가 있었습니다. 특히 **프레임당 연산 처리 속도가 최소 요구사항인 16ms를 넘어 20ms를 초과**하여 화면 버벅임(60FPS 방어 실패)이 잦았고, 무거운 캔들 객체 배열의 순회 및 잦은 수정으로 인해 **가비지 컬렉터(GC)가 주기적으로 작동하면서 화면이 멈추는 "예측 불가능한 프리징(Pause)" 현상**이 지속적으로 발생했습니다.
- **해결 방안:** 
  - 이를 기술적으로 완전히 타파하기 위해, 복잡한 코어 연산 및 레이아웃 상태(State) 로직을 모두 **고성능 시스템 언어인 Rust로 재작성하고 Wasm 바이너리로 컴파일**하여 도입했습니다.
  - V8 자바스크립트 엔진의 GC 영향권 및 객체 오버헤드에서 벗어나기 위해 캔들스틱 데이터를 JS 객체 트리 배열이 아닌 **단일 1차원 실수 배열(TypedArray, `Float64Array`)** 기반으로 촘촘하게 선형화시켰고, 이 버퍼의 메모리 포인터 주소를 Wasm과 브라우저가 직접 공유(Zero-Copy 접근)하도록 설계했습니다.
  - 이로써 내부적으로 힙(Heap) 메모리 할당/해제 스트레스를 원천 차단하여 GC로 인한 멈춤 현상을 완전히 제거하였고, `Cohen-Sutherland` 같은 최적화 라인 클리핑 알고리즘을 압도적인 Rust Native 성능과 결합한 결과, 프레임당 처리 속도를 1ms대 미만 수준으로 대폭 단축시켜 수십만 개의 데이터베이스 환경에서도 극강의 쾌적함(무한 스크롤 및 무결점 60프레임)을 성공적으로 달성해 냈습니다.

### 🔥 2) 타임프레임 스위칭 시 발생하는 그리기 도구(Drawing) 시각적 붕괴 해결
- **문제:** 사용자가 1분봉(1m) 표면에서 그린 도형이 1일봉(1d) 등 다른 타임프레임으로 전환할 시, 서로 다른 길이의 캔들 배열을 호출하게 되며 상대적인 화면 인덱스가 틀어져 선이 뷰포트 밖으로 날아가거나 찌그러지는 현상 발생 (가상화폐 거래소 데이터의 주말, 점검에 의한 빈 시간 구간으로 인한 동적 외삽 계산 오차).
- **해결 방안:**
  - 도형의 기준점을 브라우저 화면의 픽셀 인덱스 좌표계가 아닌 **절대 시간(Time) 기반 좌표계(`DrawingTimeObject`)로 분할**하여 백엔드 격인 엔진 레이어 내부에 보존했습니다.
  - 타임프레임이 변환될 때마다 해당 캔들의 고정된 간격(Interval) 상수(ex: 1분 = `60,000ms`)를 기반으로 수학적인 "정적 보간(Static Interpolation)" 과정을 재수립했습니다. 이를 통해 캔들 데이터가 실시간으로 재로딩되더라도, 차트에 그린 낙서와 수식들이 과거에 그려진 시간대의 정확한 가격 위치를 오차 없이 관통해 강제로 제자리에 맵핑(`Snap`)되도록 좌표계 복원 구조를 전면 개편했습니다.

### 🔥 3) 탭 격리(Multi-Tab) 구조와 캐싱 메모리 최적화
- **문제:** 초기 MVP 단계에서는 JS 내 전역 변수(`globalFlatData`) 단 하나로 모든 데이터를 덮어씌워서 사용하다 보니, 브라우저에서 '비트코인 1분 차트 탭'과 '이더리움 1시간 차트 탭'을 번갈아 클릭할 때마다 방금 본 데이터가 증발하고 불필요하게 바이낸스 API 서버에 동일한 데이터를 수천 개씩 중복 요청하는 자원 낭비가 있었습니다.
- **해결 방안:**
  - 각각의 탭과 쌍을 이루는 고유 식별자(`ChartContext`) 구조체를 개별적으로 생성하여 Rust 엔진 내부에 `HashMap` 형태로 철저히 격리 보관했고, JS 단에도 이와 연동되는 강력한 메모리 캐시(Cache) Map을 운영했습니다.
  - 유저 컴포넌트가 이전 탭으로 되돌아가면 비싼 네트워크 API 요청 과정을 완전히 스킵하고, 즉시 메모리에서 레이아웃 데이터만 꺼내 와 캔버스에 그립니다.
  - 바이낸스 실시간 WebSockets 연결 또한 활성화된 탭 컨텍스트별로만 수신을 동기화시켜, 수백 ms 주기로 스파이크처럼 들어오는 Live 호가를 클라이언트 로드 지연율 없이 개별적으로 관리할 수 있게 최적화했습니다.

## 📈 3. 프로젝트 성과 및 기대 효과
- **메모리 절약:** 무거운 가비지 컬렉터 추적 대상인 JS 객체 트리 대신 시스템 레벨의 버퍼 플랫 배열을 직접 조작하는 방식을 채택하여, 브라우저가 사용하는 총 가용 메모리 점유율을 획기적으로 낮췄습니다.
- **확장성 및 유지보수:** 렌더링만을 담당하는 프론트부(JS Canvas API)와 산술/상태/DB를 전담하는 코어부(Rust)가 완전히 분리된 깔끔한 패턴으로 진화했습니다. 이에 따라 추후 이동평균선(MA), 볼린저밴드 등 O(N)의 시간복잡도를 지속 요구하는 새로운 금융 보조지표 연산 기능을 추가하더라도 메인 스레드 성능 하락의 두려움 없이 엔진부에 손쉽게 탑재할 수 있게 되었습니다.
